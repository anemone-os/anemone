# 初赛冲刺策略（2026-05 至 2026-06）

## 目的

这份文档服务于一个明确目标：在初赛结束前的剩余时间内，最大化比赛通过率。

它不是长期架构规划，也不是抽象设计说明。这里记录的是接下来一个月应当如何分配工程预算、哪些事情应该停止、哪些事情必须优先，以及什么样的“脏但有效”的改动在当前阶段是可以接受的。

## 结论先行

当前阶段应当从“设计驱动”切换到“测例驱动”。

更准确地说，是切换到一种有边界的竞赛模式：

- 主目标从“把系统设计得长期优雅”切换为“让尽可能多的官方测例稳定通过”。
- 优雅性不再是当前阶段的目标，只保留为约束条件。
- 允许做兼容性特判、脚本级胶水、syscall 层放宽和面向特定 libc / testsuite 的补丁。
- 不允许做会同时打开多个核心风险面的重构。

如果继续按过去四个月的节奏推进，大概率不是“会慢一点”，而是会系统性错过比赛窗口。

## 对当前代码库的判断

### 当前代码库的优势

- 核心子系统边界相对清楚，很多实现是可推理、可维护的。
- 文档、日志和问题归档已经开始形成结构，这对快速诊断是加分项。
- 相比纯比赛导向内核，当前代码库更像一个真正准备长期演进的系统。
- 许多问题已经不是从零开始，而是 Linux 语义最后一公里的精确兼容问题。

### 当前代码库的劣势

- 测试入口还没有形成完整闭环，尤其是 LTP 仍未被当作一等目标接入。
- 开发节奏仍然偏向“先想清楚抽象”，而不是“先收敛通过率”。
- process / signal / wait / futex / poll / ioctl 这类高扇出语义仍有明显缺口。
- 当前仓库对 testsuite 状态的共享视图还不够强，容易在不同方向上分散精力。

### 相比 Quasar 的现实判断

Quasar 的优势不主要来自代码量，而来自目标聚焦。

- 它更早把比赛测例，尤其是 LTP，当成一级目标维护。
- 它愿意接受 test-specific 的路径、白名单、输出格式兼容和少量显式跳过。
- 它对“系统看起来是否漂亮”的容忍度更高，对“测例能不能过”的执念更强。

Anemone 当前并不是没有基础。相反，它的基础并不差，甚至在很多地方更像一个准备长期发展的内核。

问题在于：这不是当前时间窗口里最重要的胜负手。离初赛只剩一个月时，更重要的是把这份基础变成一个能快速收敛分数的系统。

结论是：

- 比起 Quasar，Anemone 更像“长期基础更稳”的代码库。
- 但在“一个月内冲比赛分数”这个目标上，Quasar 的工程取向目前更占优势。
- Anemone 仍然有机会，但前提是立即切换开发策略。

## 当前阶段的总原则

接下来一个月的所有任务，都应服从以下原则。

### 1. 一切工作都必须映射到测例

不再接受“这个设计以后会更好扩展”作为当前阶段的主要理由。

每项工作都必须回答：

- 它对应哪个 testsuite、哪类 case、或哪条失败日志？
- 预计会提升哪一类通过率？
- 如果它不做，未来七天内会损失什么？

如果回答不清楚，这项工作当前就不应进入最高优先级。

### 2. 停止高风险重构

以下类型的工作应默认冻结，除非它直接阻塞测例通过，并且没有更小的局部补丁路径：

- 重写调度器。
- 重做 VFS 抽象边界。
- 改造整体内存模型。
- 为“统一风格”而移动大量代码。
- 为“长远更合理”而重做 thread-group / task / signal 的整体结构。

这类工作在长期可能是正确的，但在最后一个月几乎一定会扩大不确定性。

### 3. 允许有边界的 hack

当前阶段可接受的 hack：

- 针对特定 libc 行为的兼容位放宽。
- 针对已知 case 的 syscall 层快速路径。
- runner / rootfs / 环境变量 / loader 的显式兼容胶水。
- 对官方脚本输出格式的对齐。
- 合理的 skip、TCONF、白名单和架构分流。

当前阶段不可接受的 hack：

- 悄悄改变核心不变量且没有文档登记。
- 引入 silent corruption 或资源生命周期失控。
- 一个补丁同时污染多个子系统，但没有明确退出条件。
- 为一个 case 修复，实质上把其它大量路径变成未定义行为。

### 4. 优先修高扇出语义

优先级不按模块名排，而按“影响多少测例”排。

短期最重要的是：

1. process / wait / exit / thread-group 观察语义
2. signal 交付、屏蔽、唤醒、中断和恢复语义
3. futex / clear_child_tid / robust list
4. poll / ppoll / pselect6 和阻塞唤醒
5. execve / 动态链接环境 / rootfs 布局
6. 常见 ioctl / procfs / devfs / tmpdir / 路径兼容

这些地方一旦不准，libctest、busybox、LTP、pthread、netperf、shell pipeline 会一起受影响。

### 5. 每天以“通过率变化”复盘

每天最重要的结果不是“今天改了多少代码”，而是：

- 新增通过了哪些 case / 组？
- 哪些 hang 被缩短到了可复现单项？
- 新增了哪些明确的不可支持项，并已被正确标为 skip / TCONF？

## 接下来一个月的执行模式

## 第一阶段：先建立真正的比赛闭环

时限：接下来 3 天内完成。

目标不是一次性修很多 bug，而是建立一个能稳定迭代的闭环。

### 必须完成的事情

1. 为 LTP 建立最小可用 runner

- 支持白名单。
- 支持单 case / 小批量过滤。
- 支持按 musl / glibc 分组运行。
- 输出格式尽量对齐官方脚本，避免测评侧误解析。

2. 把 testsuite 状态写成共享面板

- 明确 basic、busybox、libctest、LTP、netperf、iozone、cyclictest 当前状态。
- 明确每个 testsuite 当前是“未接线 / 可运行 / 部分通过 / 大量 hang / 已稳定”。
- 不再让“我们感觉差不多”取代共享事实。

3. 建立最小种子白名单

- 先不要追求 LTP 全量。
- 先选 20 到 30 个覆盖高扇出语义的种子 case。
- 目标是让这些 case 变成每日回归基线。

4. 让 rootfs 和运行环境可复现

- 明确 musl / glibc 的 loader、`LD_LIBRARY_PATH`、`PATH`、`LTPROOT`、`/tmp`、`/proc`、`/dev`。
- 避免“环境没起来”和“内核语义不对”混在一起。

### 第一阶段的验收条件

- 单个 LTP case 可以稳定跑起来并拿到内部 summary。
- 小批量白名单可以重复运行。
- 每条失败至少能落到“语义问题 / 环境问题 / 预期不支持”之一。

## 第二阶段：按高扇出簇收口

时限：第 1 周到第 3 周。

这一阶段不要按子系统所有权切，而要按“失败簇”切。

### 簇 A：process / wait / exit / thread-group

目标：先把所有依赖 fork / wait / exec 的测试稳定下来。

优先修复点：

- wait4 / waitid 的返回与观察语义。
- 线程组退出与组长可见状态。
- 子进程回收与 parent 观察事件。
- `WNOHANG`、`WUNTRACED`、`WCONTINUED` 的最低可用语义。
- exit、signal-exit、group-exit 的一致性。

为什么优先：

- 这是 shell、busybox、libctest、LTP process 类 case 的共同基础。
- 这里一旦不准，会反复出现 `ECHILD`、错误退出码、永远等不到子进程等问题。

### 簇 B：signal

目标：让用户态的“安装信号处理器、屏蔽、唤醒、中断阻塞 syscall”具备 Linux 可接受语义。

优先修复点：

- `rt_sigaction` 兼容位，尤其是常见 libc 标志位。
- `rt_sigprocmask` 的解除屏蔽后唤醒。
- `rt_sigtimedwait` 与普通异步交付的关系。
- `sigaltstack`、`tgkill`、线程定向信号。
- stop / continue 的最小可用语义。

为什么优先：

- pthread、超时、wait 中断、libctest 和大量 LTP case 都依赖这一层。

### 簇 C：futex / pthread 基础

目标：把用户态线程同步从“偶尔能跑”变成“可预测”。

优先修复点：

- `FUTEX_WAIT` / `FUTEX_WAKE` 基础语义。
- `WAIT_BITSET` / `WAKE_BITSET`。
- clear_child_tid 唤醒。
- robust list 基础清理路径。
- `EINTR`、`ETIMEDOUT`、`EAGAIN` 的边界语义。

为什么优先：

- 如果 futex 不稳，glibc / musl 的大量线程用例会把错误表象扩散到 wait、signal、exec、stdio 等其它表面。

### 簇 D：poll / iomux / pipe / socket 阻塞语义

目标：先让阻塞行为“像 Linux”，再谈实现是否优雅。

优先修复点：

- `ppoll` / `pselect6` 参数兼容与 ready count。
- pipe 和 socket 的可读可写判定。
- 信号中断阻塞 syscall 的返回语义。
- 超时、空 fd 集、零超时、被唤醒后重新探测。

为什么优先：

- netperf、busybox shell pipeline、部分 LTP、libctest 都会集中命中这一层。

### 簇 E：execve / loader / rootfs / 常见文件接口

目标：降低“环境没起来”对诊断的干扰。

优先修复点：

- 动态加载器路径与 `libc.so` / `ld.so` 链接布局。
- rootfs 中 `/bin`、`/usr/bin`、`/sbin`、`/proc`、`/tmp`、`/dev` 的最小兼容。
- 常用 `openat`、`stat`、`read`、`write`、`lseek`、`ioctl` 缺口。
- LTP 常用临时目录与 data file 查找路径。

为什么优先：

- 很多“内核 bug”其实只是 loader 或 rootfs 兼容问题。

## 第三阶段：扩白名单，不扩理想

时限：最后 1 周。

这时的目标不再是做新设计，而是扩大通过面并压缩不确定性。

### 要做的事

- 从种子白名单逐步扩到更大白名单。
- 识别最慢、最容易 hang 的 case，并逐个单项化。
- 区分真正失败、环境缺口、架构差异和合理 skip。
- 为 RISC-V 和 LoongArch 拆开统计，不要求二者完全同步推进。

### 不要做的事

- 不要在最后一周开启新的大重构。
- 不要为了“统一实现”推倒已经能跑通的竞赛路径。
- 不要为了清理技术债去动已经稳定的测试入口。

## 每周节奏建议

## 第 1 周

- 接通 LTP 最小 runner。
- 建立 testsuite 状态面板。
- 选定种子白名单。
- 用现有日志把失败粗分为 wait、signal、futex、poll、fs、env 六类。

## 第 2 周

- 集中打 process / wait / signal。
- 目标是让 busybox、libctest、LTP 中依赖 fork / wait / signal 的用例显著改善。

## 第 3 周

- 集中打 futex / poll / pipe / socket / timeout。
- 目标是减少 hang、提高 pthread 和网络相关测例的稳定性。

## 第 4 周

- 集中做白名单扩张、超时清理、环境兼容和架构分流。
- 所有改动都以稳定性和通过率为目标，不做大设计。

## 每日工作流建议

每一天按同一个节奏执行。

### 上午

- 选一个失败簇。
- 只挑 3 到 5 个代表 case。
- 让这些 case 都能单独复现。

### 下午

- 只修这一簇，不跨簇跳题。
- 每次改动后立刻跑最小复现。
- 不在同一天同时开 process、signal、poll 三条线。

### 晚上

- 更新 testsuite 状态表。
- 更新开发日志。
- 记录新增 hack 的目标 case、原因和退出条件。

## 允许做的“竞赛模式”改动

为了避免团队对“是否太脏”反复犹豫，当前阶段明确允许以下做法：

- 在 syscall 层做 Linux 兼容位的放宽和参数兼容。
- 为常见 testsuite 输出格式做对齐。
- 为特定 libc 版本或行为差异做显式兼容。
- 针对已知不支持项，尽早做合法的 skip / TCONF，而不是让它随机 hang。
- 先在用户态 runner 和 rootfs 层解决环境问题，而不是一开始深入内核重构。

这些做法只有两个前提：

- 必须能说清楚服务于哪类测例。
- 必须在文档或注释里留下退出条件。

## 当前阶段必须停止的事情

以下事情如果继续做，会明显拉低比赛期望收益：

- 为了统一抽象而重排目录或接口。
- 为了未来可能支持更多平台而提前泛化。
- 对尚未成为比赛阻塞项的模块做性能洁癖式优化。
- 在没有 runner 和白名单的前提下尝试全量 LTP。
- 在没有共享状态面板的前提下凭感觉决定下一步。

## 成功标准

到初赛结束前，成功不应被定义为“代码终于优雅了”，而应定义为以下事实尽可能多地成立：

- 官方 testsuite 的通过面持续扩大。
- LTP 已经有稳定 runner、白名单和单项复现能力。
- 大部分失败都已从“模糊 hang”转为“明确且可定位的语义缺口”。
- 团队不会再因为是否应该重构而浪费一整天。
- 所有 hack 都是局部、已登记、可解释的，而不是失控扩散的。

## 最后一句话

当前阶段最重要的不是证明自己当初的设计路线完全正确，而是先活过比赛。

如果初赛目标是第一优先级，那么现在最正确的工程选择就是：

- 接受局部不优雅；
- 拒绝全局失控；
- 把所有精力转成可度量的通过率提升。

这不是放弃技术标准，而是把技术标准切换到当前阶段真正需要的那一种。
