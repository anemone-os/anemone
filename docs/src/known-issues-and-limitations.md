# 已知问题与当前限制

## 一个薄的活动登记册，两种条目类型

对当前仍然重要、并且会影响诊断或验证判断的事项，维护一个短小的活动登记册，并把每个条目标记为以下两类之一：

- `Issue`：本来应该工作，但当前没有正确工作的行为。
- `Limitation`：当前有意缺失、尚未完成、或暂不支持的能力。

这个区分很重要，因为 agent 需要知道自己面对的是回归，还是当前可接受的缺口。

## 与 GitHub issues 的分工

GitHub issues / PR 负责：

- 工作项规划与验收条件；
- 讨论过程、备选方案和长调查线程；
- 跨阶段推进中的 ownership 与状态流转。

活动登记册只负责：

- 当前仍生效的操作性事实；
- 会改变当前诊断、验证或兼容性判断的事项；
- 一眼可扫描的当前状态卡片。

不是每个 GitHub issue 都应该进入登记册。只有当一个问题或限制已经足够明确，并且它会影响今天的开发判断时，才值得在这里保留一条短记录。

如果已经存在规范的 GitHub issue / PR，应在条目中链接它，而不是把整段讨论复制到登记册里。

## 字段门槛

活动登记册默认采用最小必填字段。

两类条目的共同必填字段：

- `ID`
- `Type`
- `Status`
- `Area`
- `Owner`
- `Last Verified`
- `Exit Condition`
- `Related`

`Issue` 额外必填：

- `Symptom / Trigger`
- `Impact`

`Limitation` 额外必填：

- `Summary`

按需字段：

- `Severity`：确实需要快速分级时再写。
- `Workaround`：存在实际可用的临时路径时再写。
- `First Seen`：起源时间会影响判断时再写。
- `Tracker`：已有规范的 GitHub issue / PR 或其他长期跟踪入口时再写。

## 生命周期

- 当问题已经可复现，或限制已经被明确接受，并且它会影响当前开发判断时，加入登记册。
- 如果某个事项目前只需要排计划、收集讨论或排队等待，不进入登记册，留在 GitHub issue / PR。
- 每次重新确认问题仍存在时，更新 `Last Verified`。
- 当问题修复、过时、或被替代后，将条目从登记册中移除；长期历史保留在开发日志和 GitHub issue / PR 中。
- 每个条目至少链接一条开发日志；如果存在规范的 tracker，也应链接它。

实际条目维护在活动登记册章节中，并按“开放问题”与“当前限制”分开存放。

## 范围纪律

这个登记册记录的是“当前仍然生效的操作性事实”，而不是项目史上出现过的所有 bug，也不是另一套 backlog。

一个问题修复后，长期历史应保留在开发日志里；活动登记册本身应始终短到可以快速扫描。
