# 2026-06-08 - Fanotify

**Status:** Active
**Owners:** doruche, Codex
**Area:** fs / VFS / syscall ABI / task-fd / iomux / LTP
**RFC:** [RFC-20260604-fanotify](../../rfcs/fanotify/index.md)
**Current Phase:** Gate B closed; D4 not started

## Scope

本事务跟踪 fanotify 子系统按 RFC gate 的 staged 实现。Gate A 已合并 D1 task/fd
通用能力与 D2 `fs::fanotify` owner module / group fd，使 Stage 0 + Stage 1 成为第一个
用户可见 gate。当前阶段推进 D3 mark registry 与 `fanotify_mark()`，完成后进入 Gate B
review；D4 path-fd read 和 D5 VFS hook 继续按依赖顺序后置。

本事务按 [Agent 编排建议](../../rfcs/fanotify/backgrounds/agent-orchestration.md)
推进，不按文档阶段机械拆分 worker。实现顺序以依赖拓扑为准：

- D1：task/fd 通用能力；
- D2：fanotify owner module、UAPI parser、group fd、bounded queue；
- Gate A review：确认 Stage 0 + Stage 1 合并 gate 闭合；
- D3：mark registry、target identity、basic mark parser 和 add/remove/flush；
- D4+ 后续阶段暂不在本轮启动。

非目标：

- 不在 D3 实现真实 VFS enqueue、path-fd event fd 或 FID/name ABI。
- 不在 Gate A 宣称 LTP fanotify 基础用例通过。
- 不运行 QEMU / LTP，除非用户后续明确要求；运行态证据默认由用户提供。
- 不通过 fanotify 私有 fd number 状态、private nonblock mirror 或 fd-close bridge 来替代
  task/fd 通用能力。

## Invariants

- `fs::fanotify` 是 group、queue、registry、mark record、matching 和 file ops private
  state 的 owner；模块外只能通过 typed facade 交互。
- `fanotify_init()` 的用户可见成功只能在 Stage 0 + Stage 1 合并 gate 记录。
- group fd nonblock read 必须读取当前 opened file description 的 `FileStatusFlags`，
  不能固化成 fanotify private state。
- fd reservation / commit / rollback、opened-description final-release callback 和
  notification suppression 属于 task/fd 通用能力，不能由 fanotify 维护并行真相源。
- fanotify group fd 必须提供完整 `FileOps` vtable：read / poll / optional ioctl 走
  group fd file ops，seek / positioned I/O / read_dir fail closed。
- queue cap 和 overflow sentinel 必须早于真实 VFS enqueue。
- 每个 worker 只能写入编排文档指定 write set；需要扩大 write set 时必须停止上报。

### 2026-06-08 - Agent 1 D1 task/fd 通用能力

**Phase:** Gate A / D1

**Change:** Agent 1 在 task/fd 层落下 fanotify 下游共享能力，不新增 fanotify policy：

- `read()` / `readv()` 在 ordinary kernel-buffer fallback 前尝试 opened-description
  generic read-user hook；hook 接收当前 opened file description 的 `FileStatusFlags`、
  当前 task userspace 和一个或多个用户缓冲段。`pread*` / positioned read 不进入该 hook。
- `FilesState` 增加 fd reservation / commit / rollback。reserved slot 占用 allocator
  bitmap，但在 commit 前 `fds[fd] == None`，因此普通 fd allocation 不会复用，`get_fd()`、
  close 和 fd snapshot 也不可观察；commit 只发布已准备好的 `FileDesc`，rollback 幂等清理。
- opened-description final-release 使用 fd-table published-slot refcount，而不是
  `Arc<FileDesc>` transient ref 或 fd-number close bridge；dup/fork 共享同一个 `ProcFile`，
  最后一个 published slot 关闭后在 fd-table 锁外运行 callback。
- notification suppression 是 generic opened-description marker，task/fd 核心只保存和暴露
  capability，不检查 fanotify concrete type。

**Review:** D1 未修改 `fs::fanotify`，也未新增 fanotify-specific downcast。普通 `open_fd()`
继续使用 empty description ops；需要特殊 opened-description 行为的下游必须显式调用
`open_fd_with_description_ops()` 或用 `FileDesc::new_opened()` 配合 fd reservation commit。
final-release callback 在锁外执行，避免后续 fanotify teardown 受 fd-table 锁序限制。

**Validation:** `git diff --check` 通过。`just fmt kernel --check` 未通过，但失败包含既有
非 D1 文件格式噪声（`lwext4-rust`、generated `kconfig_defs.rs` / `platform_defs.rs`、
`task/sig`）；Agent 1 仅按 formatter 建议手工收敛了本次 `task/files.rs` 格式。sandbox
内 `just build` 在 `lwext4_rust` C 编译触发已知 `Bad system call`。非 sandbox `just build`
越过该问题后，当前失败来自 Agent 2 write set 的 `fs/fanotify/api/{init,mark}.rs` 缺少
`TryFromSyscallArg` import；未暴露 D1 代码错误。

**Next:** Agent 2 修复 owner module 编译后，合流运行 Gate A review；重点检查 read-user hook
是否被 fanotify group fd 使用、`FAN_NONBLOCK` 是否读取 current `FileStatusFlags`、reservation
commit/rollback 是否覆盖 path-fd read 提交协议的需要，以及 final-release 是否支撑 group
semantic teardown。

### 2026-06-08 - Gate A review 修复

**Phase:** Gate A / review

**Review:** 只读 reviewer 针对 D1+D2 diff 和 RFC Gate A 条件发现三个 Keter：

- dead group read state 与 live-empty state 混在一起，导致 nonblock read 可能在 group dead
  后返回 `EAGAIN` 而不是 EOF。
- fanotify `poll()` 把 `HANG_UP` 按 caller interests 过滤，`ppoll(POLLIN)` 可能看不到
  close/dead readiness。
- `fanotify_mark()` 对 known/deferred mask 走 `NotYetImplemented` placeholder，映射成
  `ENOSYS`，不符合 RFC 稳定 fail-closed errno matrix。

**Change:** Controller 在 Gate A write set 内修复：

- `FanGroup::pop_read_state()` 返回 `FanReadState::{Event, Empty, Dead}`，`fanotify_read_user()`
  先把 `Dead` 映射为 EOF，再对 live-empty + current `NONBLOCK` 返回 `EAGAIN`。
- blocking read waiter 从 `yield_now()` busy loop 改成 waiter-owned `Latch`；read trigger 在
  group lock 下注册，并在注册后同锁重扫 queue/dead，enqueue/dead detach 后在锁外 trigger。
- `FanQueue::poll()` 只按 interests 控制 `READABLE`，dead `HANG_UP` 无条件进入 revents，
  因此 dead group 不会重新 arm 一个不可触发 waiter。
- registry / mark placeholder 返回 `SysError::InvalidArgument`，并用注释记录 Gate A 已有
  syscall entry 但 mark storage 语义仍延后到 D3。
- `fs/fanotify/mod.rs` 对 Gate A 暂未被外部消费的 typed facade re-export 添加局部
  `allow(unused_imports)`，保留 RFC 约定的 owner-module facade，同时消除 fanotify 自身
  build warning。

**Residual Risk:** Reviewer 记录一个 Euclid：vectored metadata copy 在 D4 path-fd submit
协议前仍可能在后续 segment fault 时留下已 copy 的字节。Gate A synthetic `FAN_NOFD`
路径不把它作为 blocker；D4 必须在 event fd 安装、metadata copyout 和 rollback 协议中
系统性处理。

**Recheck:** Reviewer 复核确认三个 Keter 已关闭，结论为：Gate A 无 Apollyon / Keter
blocker。Euclid 残留仍按 D4 path-fd read submit gate 处理，不阻塞本 Gate。

### 2026-06-08 - Agent 3 D3 mark registry 与 fanotify_mark

**Phase:** Gate B / D3 implementation

**Write Set Expansion:** 用户批准 D3 write set 仅增加
`anemone-kernel/src/fs/fanotify/file.rs`。原因是 `fanotify_mark()` 需要从 group fd 的
owner-module private state 取得 `Arc<FanGroup>`，原 D3 write set 若在 syscall 层绕行
会迫使 `api/mark.rs` downcast private state，违反 `fs::fanotify` owner-module contract。
扩展范围限定为 `file.rs` 内部窄 helper `group_from_file(&File) -> Result<Arc<FanGroup>,
SysError>`；downcast 仍只存在于 `fs/fanotify/file.rs`，syscall / VFS / task-fd / procfs
层不访问 fanotify concrete private state。contract/gate 影响：保持 owner-module facade
边界，允许 D3 registry 绑定 group id / generation / weak handle；不触碰 D4 path-fd read、
VFS hook、task/fd reservation、read/write syscall 或 procfs。

**Change:**

- `FanTarget::{Inode, Mount, Filesystem}`、`FanTargetKey::{Inode, Mount, SuperBlock}` 和
  `FanPathKey { mount, dentry }` 落到 `types.rs`，集中封装 target/path identity。
- `FanMarkRecord` 成为 mark state 唯一 owner，保存 event mask、ignored mask、
  `ignored_survives_modify`、target strong ref、`target_dead`、group id/generation 和 weak
  group handle；target map 和 group cleanup list 只保存 `MarkHandle`。
- `registry.rs` 建立全局 registry lock、slot map 和 `marks_by_target`，实现
  ADD / REMOVE / FLUSH 的同锁线性化策略。group teardown 先按 registry -> group 顺序标记
  dead 并删除 mark records，释放 registry lock 后再清 group queue / 触发 waiters，避免
  registry lock 包住后续可能变重的 queue drop。
- `group.rs` 增加 group generation、mark cleanup handle list 和 registry-owned
  teardown helpers；group state 不复制 mask、target refs、target_dead 或 mark generation。
- `fanotify_mark()` 在进入 registry 前完成 command/target/mask validation，显式处理
  `FAN_MARK_INODE == 0`，支持 inode / mount / filesystem target、`ONLYDIR`、
  `DONT_FOLLOW`、`EVENT_ON_CHILD`、`ONDIR` 和 legacy `FAN_MARK_IGNORED_MASK` /
  `FAN_MARK_IGNORED_SURV_MODIFY`。`FAN_MARK_IGNORE`、`FAN_MARK_EVICTABLE`、permission、
  FID/name/dirent/self/metadata、FS error 和 exec-open masks 均返回稳定 `EINVAL`，不进入
  registry。mount / filesystem mark 需要 `CAP_SYS_ADMIN`。
- path resolution 遵守 `pathname == NULL` 时以 `dfd` 作为目标、相对路径以 directory `dfd`
  为 base、`DONT_FOLLOW` 不跟随最后一级 symlink、`ONLYDIR` 对最终 target 返回 `ENOTDIR`。

**Review:** 本地自审未发现 mark state 第二真相源、模块外 downcast、fd-number state、
tracking issues 进度表化或 D4 范围越界。`FAN_MARK_IGNORE` 新语义未被 legacy ignored-mask
路径接受。

**Validation:**

- `git diff --check`：通过。
- `just build`：通过；build 输出只剩既有 `anemone-kernel/src/sync/mono.rs` unused-import
  warning。
- QEMU / LTP：未运行，符合用户要求。

**Residual Risk:** D5 matching 尚未接入，`ignored_survives_modify == false` 的 modify 后清除
只能等 VFS `FAN_MODIFY` hook 调用 registry matching/ignore 逻辑时验证。首批 target
lifecycle 仍按 RFC 接受的 strong-ref + pre-unmount flush / mark-dead 口径；完整 umount /
filesystem kill 清理由后续 VFS lifecycle hook gate 负责。`fanotify04`/手工
