# 2026-06-08 - Fanotify

**Status:** Active
**Owners:** doruche, Codex
**Area:** fs / VFS / syscall ABI / task-fd / iomux / LTP
**RFC:** [RFC-20260604-fanotify](../../rfcs/fanotify/index.md)
**Current Phase:** Gate A setup

## Scope

本事务跟踪 fanotify 子系统按 RFC gate 的 staged 实现。首个实现目标是 Gate A：
合并 D1 task/fd 通用能力与 D2 `fs::fanotify` owner module / group fd，使 Stage 0
+ Stage 1 成为第一个用户可见 gate。

本事务按 [Agent 编排建议](../../rfcs/fanotify/backgrounds/agent-orchestration.md)
推进，不按文档阶段机械拆分 worker。实现顺序以依赖拓扑为准：

- D1：task/fd 通用能力；
- D2：fanotify owner module、UAPI parser、group fd、bounded queue；
- Gate A review：确认 Stage 0 + Stage 1 合并 gate 闭合；
- D3+ 后续阶段暂不在本轮启动。

非目标：

- 不在 Gate A 实现 mark registry、真实 VFS enqueue、path-fd event fd 或 FID/name ABI。
- 不在 Gate A 宣称 LTP fanotify 基础用例通过。
- 不运行 QEMU / LTP，除非用户后续明确要求；运行态证据默认由用户提供。
- 不通过 fanotify 私有 fd number 状态、private nonblock mirror 或 fd-close bridge 来替代
  task/fd 通用能力。

## Invariants

- `fs::fanotify` 是 group、queue、registry、mark record、matching 和 file ops private
  state 的 owner；模块外只能通过 typed facade 交互。
- `fanotify_init()` 的用户可见成功只能在 Stage 0 + Stage 1 合并 gate 记录。
- group fd nonblock read 必须读取当前 opened file description 的 `FileStatusFlags`，
  不能固化成 fanotify private state。
- fd reservation / commit / rollback、opened-description final-release callback 和
  notification suppression 属于 task/fd 通用能力，不能由 fanotify 维护并行真相源。
- fanotify group fd 必须提供完整 `FileOps` vtable：read / poll / optional ioctl 走
  group fd file ops，seek / positioned I/O / read_dir fail closed。
- queue cap 和 overflow sentinel 必须早于真实 VFS enqueue。
- 每个 worker 只能写入编排文档指定 write set；需要扩大 write set 时必须停止上报。
