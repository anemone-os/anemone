# ANE-CHG-20260724-fionbio

**Type:** Small Feature / Linux ABI Compatibility
**Status:** Completed
**Date:** 2026-07-24
**Authors:** doruche, Codex
**Area:** VFS / ioctl / fcntl / file status flags / user-test

## Problem

Anemone 的 `ioctl` 路径尚未处理 Linux `FIONBIO`。用户态因此不能通过
`ioctl(fd, FIONBIO, int *)` 修改 opened file description 上的 `O_NONBLOCK`，只能使用
`fcntl(F_SETFL)`。如果直接把该命令下沉到 `FileOps::ioctl`，后端就会获得修改
opened-description status truth 的第二条路径，并可能与 `F_SETFL` 并发覆盖彼此更新。

原有验证选择也需要收窄：LTP 20240524 没有直接覆盖 `FIONBIO` 的测例；纯 KUnit
只能证明候选状态的拒绝与并发提交规则，不能证明 syscall pointer ABI、`F_GETFL`
可见性、pipe 非阻塞行为和 dup 共享语义。

## Scope

本轮只完成 Linux `FIONBIO` 的 generic fd 语义：

- 从用户 `int *` 读取开关值，零表示清除，任意非零值表示设置 `O_NONBLOCK`；
- 在 opened file description owner 上更新状态，并保留其它 status flags；
- 让 `F_SETFL` 与 `FIONBIO` 复用同一条 backend validation 和 compare-and-commit 路径；
- 增加可继续容纳其它 ioctl ABI 回归的独立 `ioctl-test`，并接入 pretest rootfs 和
  `user-test`。

本轮不新增通用 unsafe 用户态 `ioctl` wrapper，不扩展其它 ioctl command，不补齐
`openat` 的剩余 flag 差异，也不实现 `O_DIRECT`、`O_SYNC`、`O_DSYNC` 或 `O_NOATIME`
的完整后端语义。

## Solution

`FIONBIO` 在 generic syscall adapter 完成 fd lookup 和 `O_PATH` 拒绝后截获，不进入
`FileOps::ioctl`。这是 opened file description 的 status mutation，而不是具体设备的
file-object command。adapter 只负责解析 `int *` ABI，真正提交由 `FileDesc` 的窄接口
完成。

状态更新采用 snapshot、锁外 backend validation、compare-and-commit 重试：先读取当前
flags并形成候选值，在不持有 status lock 时调用 side-effect-free
`check_status_flags()`；重新加锁后只有 snapshot 未变化才提交，否则基于新状态重算。
这样既不把 task/files 锁序带入后端，也避免并发 `F_SETFL` / `FIONBIO` 丢失更新。

验证采用三层而不是宽泛矩阵：colocated KUnit 覆盖提交不变量；`ioctl-test` 覆盖真实
syscall ABI 和用户可见行为；现有 LTP signal/wait 清单只作为同一镜像上的回归背景，
不作为 `FIONBIO` 功能证明。单独建立 `ioctl-test` 是为了给后续 ioctl ABI 回归一个明确
入口，但本轮不预先抽象 command registry 或暴露 raw pointer API。

## Change

- `anemone-abi` 增加 asm-generic/Linux `FIONBIO` command 常量。
- generic `sys_ioctl` 读取用户 `i32`，按零/非零更新 `O_NONBLOCK`，并保持既有 bad-fd、
  `O_PATH` 与 user-pointer error 边界。
- `FileDesc` 增加统一 status update helper；`F_SETFL` 和 `FIONBIO` 分别通过窄接口提交，
  backend 拒绝时不改变原状态，并发修改时重新计算。
- `anemone-rs` 增加 typed `ioctl_set_nonblocking()` 和测试所需的 `dup()` wrapper，没有
  暴露通用 unsafe ioctl surface。
- 新增 `anemone-apps/ioctl-test`，当前包含 set/clear roundtrip、其它 flag 保留、空 pipe
  `EAGAIN`、dup 共享 opened-description 状态和 bad-fd `EBADF` 三组场景；RV64/LA64
  pretest rootfs 均安装该程序，`user-test` 在 LTP 前运行。

## Validation

- `just fmt kernel`、`just fmt ioctl-test`、`just fmt user-test` 通过。
- `just app build ioctl-test --arch riscv64` 与
  `just app build ioctl-test --arch loongarch64` 通过。
- `just build --preset qemu-virt-rv64-pretest-release` 通过。
- RV64 repository wrapper 启动真实 guest；254 项 KUnit 全部通过，其中新增的 backend
  rejection 与 concurrent retry 两项均通过。
- guest 中 `fionbio-roundtrip`、`fionbio-shared-description`、`fionbio-bad-fd` 三项
  `ioctl-test` 全部通过，并输出 `ioctl-test: all cases passed`。
- 同次运行的 LTP whitelist 为 attempted=120、passed=106、failed=10、skipped=4；其中
  没有直接 `FIONBIO` 测例，因此该结果只记录为回归背景，不计入功能证明。
- 最终 RV64 repository wrapper 在测试清单结束并输出
  `user-test-heartbeat: control_pipe_closed` 后正常返回 exit 0；日志未发现 panic 或
  deadlock。guest 没有输出额外的文字化 shutdown 行，因此证据结论限于 wrapper/QEMU
  正常退出，不声称观察到了独立的关机消息。
- LA64 只完成 `ioctl-test` 交叉编译；LA64 kernel build、QEMU runtime、硬件和外部
  ioctl conformance suite 均未运行。

## Tracking Issues

### CHG-001 - 用户指针错误与负非零值未进入 typed runtime case

**Status:** Deferred
**Severity:** Safe

**Issue:** 当前 typed wrapper 总是传入有效 `i32`，用户态 runtime case 没有直接制造
`EFAULT`，也没有单独传入负非零值。

**Resolution:** kernel 路径使用既有 `UserReadPtr<i32>`，并按 `value != 0` 解释开关；
本轮不为了两个低增量 case 暴露通用 unsafe ioctl wrapper。后续若 `ioctl-test` 引入受控
raw-pointer test helper，可补入该边界。

## Risk / Follow-up

- `ioctl-test` 是可扩展的用户态 ABI 测试入口，不意味着所有 ioctl 都应走 generic
  status path；后续 command 仍需按 fd-owned、opened-description-owned 或 backend-owned
  语义分类。
- current limitation 中 open/status flags 的 legacy unknown-bit、`O_PATH` 回显以及
  sync/direct/noatime 真实语义仍然存在，本轮只增加 `FIONBIO` mutation 入口。
- LA64 runtime 仍是未运行边界，RV64 wrapper exit 0 的证据不得外推。
