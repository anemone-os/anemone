# ANE-CHG-20260614-waitid

**Type:** Small Feature / Syscall / LTP Compatibility
**Status:** Completed
**Date:** 2026-06-14
**Authors:** doruche, Codex
**Area:** task / wait / syscall ABI / process group / LTP

## Problem

`waitid` syscall 之前没有接入。当前 register 已经把进程组与会话 stage-1 限制记录为
缺少 `waitid` 的 P_PID / P_PGID / P_ALL 基础语义，LTP waitid 测例也因此无法进入
wait group。

本轮问题适合作为小迭代处理：内核已经有 exited-child wait 能力和 `wait4` 的基础选择
语义，目标是把已经存在的能力接到 Linux `waitid` ABI 上，而不是扩展完整 job-control。

## Scope

本轮只接入 exited-child waitid bridge：

- rv64 / la64 新增 `SYS_WAITID = 95`。
- 支持 `P_ALL`、`P_PID`、`P_PGID`。
- 支持 `WEXITED`、`WNOHANG`、`WNOWAIT`，并接受既有 Linux wait compat bits。
- `siginfo_t` 只写 waitid 可见的 `si_signo`、`si_errno`、`si_code`、`si_pid`、
  `si_uid`、`si_status` 字段。
- 可选 `rusage` 复用当前 stage-1 task CPU accounting。
- LTP wait group 只启用当前支持的 `waitid01` 至 `waitid06`、`waitid09` 和
  `waitid11`。

本轮不实现 stopped / continued child reporting、`P_PIDFD` wait、core-dump reporting、
controlling tty、foreground/background process group 或 terminal job-control 信号。
`waitid07`、`waitid08` 和 `waitid10` 继续留在 wait group 注释中，作为明确的后续边界。

## Solution

把 wait-family syscall 统一放到 `task/api/wait/` 目录下。`wait4` 和 `waitid` 共用
exited-child scan / reap helper，syscall 层只负责各自 ABI parser、返回值和 userspace
写回格式。

`waitid` 在 `WSTOPPED` / `WCONTINUED` 或 `P_PIDFD` 路径上 fail closed 并打日志，避免
把未实现的 stopped/continued 状态或 pidfd identity 伪装成成功。`WNOWAIT` 映射为 peek，
普通 waitid 继续执行 reap；reap race 发生时重新扫描 child list。

`siginfo_t` 不整块复制内核临时结构，而是逐字段写入 Linux ABI 明确由 waitid 暴露的字段。
这样避免 Rust 端 UAPI union padding / layout 细节变成 userspace 可见行为。

## Change

- `anemone-abi` 新增 waitid syscall number、`P_ALL` / `P_PID` / `P_PGID` / `P_PIDFD`、
  `CLD_*` 与 `__W*` wait compat bits。
- `task/api/wait4.rs` 拆到 `task/api/wait/` 目录；`mod.rs` 承载共享 wait core，
  `wait4.rs` 和 `waitid.rs` 保持 syscall thin layer。
- `waitid` 对 `P_ALL` / `P_PID` / `P_PGID` 复用 exited-child wait core，返回值保持
  Linux waitid 的 `0`，selected child 通过 `siginfo_t` 报告。
- `waitid` 的 signal exit 当前报告为 `CLD_KILLED`；内核尚未跟踪 core-dump 状态，不猜测
  `CLD_DUMPED`。
- `anemone-apps/user-test/ltp/groups/wait.txt` 启用已支持的 waitid 测例，并在不支持项旁
  标出依赖原因。

## Tracking Issues

### CHG-001 - waitid siginfo union writeback

**Status:** Neutralized
**Severity:** Apollyon

**Issue:** 如果把内核侧 `SigInfo` 临时对象整块写给 userspace，Rust 端 union padding、
未初始化字节或结构布局差异可能变成 ABI 可见行为。

**Resolution:** `waitid` 只逐字段写入 Linux waitid 用例可见且 ABI 明确的字段：
`si_signo`、`si_errno`、`si_code`、`si_pid`、`si_uid`、`si_status`。

### CHG-002 - concurrent reap stale scan

**Status:** Neutralized
**Severity:** Keter

**Issue:** 多线程同时等待同一个 exited child 时，扫描阶段选中的 child 可能在 reap 前被
另一个 waiter 回收。如果此时沿用旧扫描状态，可能错误返回或丢失重新检查机会。

**Resolution:** `try_reap_child()` 返回 `None` 时丢弃本轮扫描状态并重新进入 wait loop。
后续重新判断是否还有 matching child、是否应返回 `ECHILD`，或是否继续等待。

### CHG-003 - non-exit waitid states

**Status:** Deferred
**Severity:** Euclid

**Issue:** `waitid07` / `waitid08` 依赖 stopped / continued child reporting；`waitid10`
依赖 core-dump reporting 以及 core pattern / rlimit 相关 setup；`P_PIDFD` wait 依赖
pidfd file object 与稳定 identity。

**Resolution:** 本轮明确作为 exited-child bridge 收口。剩余能力继续记录在当前限制中，
wait group 仅启用已支持测例。

## Risk / Follow-up

- `waitid07` / `waitid08` 需要 job-control stopped / continued wait reporting。
- `waitid10` 需要 core-dump 状态跟踪，并补齐 `/proc/sys/kernel/core_pattern` 与
  `getrlimit(RLIMIT_CORE)` 相关路径。
- `P_PIDFD` wait 需要 pidfd 文件对象、exit notification 与稳定 pidfd identity。
- wait group 中 `wait402` / `wait403` 仍受 `/proc/sys/kernel/pid_max` 与
  `/proc/sys/kernel/tainted` 缺失影响。
