# ANE-CHG-20260613-eventfd

**Type:** Small Feature / Syscall / Anonymous FD
**Status:** Completed
**Date:** 2026-06-13
**Authors:** doruche, Codex
**Area:** VFS / FileOps / iomux / syscall ABI / LTP

## Problem

rv64 / la64 的 asm-generic syscall table 之前没有 `eventfd2` 入口。LTP
`eventfd2_*` 会直接调用 syscall number 19，普通 `eventfd01..05` 也会经 musl
`eventfd()` wrapper 走 `SYS_eventfd2`，因此当前失败主要表现为 syscall 缺失或
`CONFIG_EVENTFD` fixture 缺失，而不是 counter 语义本身的失败。

eventfd 适合作为小迭代处理：匿名 inode、opened-description status snapshot、
`FileMode::STREAM`、fanotify notification suppression 和 latch poll 注册都已经存在。
本轮风险集中在不要把 eventfd 塞进普通文件 cursor 语义，也不要为了 eventfd 引入完整
provider/AIO 子系统。

## Scope

本轮实现用户可见 `eventfd2(count, flags)` fd：

- rv64 / la64 按 asm-generic 定义新增 `SYS_EVENTFD2 = 19`。
- 新增 Linux `EFD_SEMAPHORE`、`EFD_CLOEXEC`、`EFD_NONBLOCK` 常量。
- 新增 eventfd anonymous `STREAM` file 后端，counter state 是唯一行为真相源。
- 覆盖 `read` / `write` counter 语义、blocking / nonblocking、`EFD_SEMAPHORE`、
  `poll` / `select` readiness、dup / fork 后共享 opened file description state。
- `EFD_CLOEXEC` 映射到 fd flags，`EFD_NONBLOCK` 映射到 opened-description status，
  `EFD_SEMAPHORE` 只保存在 eventfd 私有 state。
- LTP fixture 声明 `CONFIG_EVENTFD=y`，并提供 eventfd group 入口。

本轮不实现 legacy `eventfd(count)` syscall、AIO/eventfd provider API、eventfd overflow
provider 语义、`/proc/<pid>/fdinfo` eventfd 字段、epoll 专门路径、scalar I/O 尾部坏地址
兼容、跨 iovec 聚合 8 字节读写，或 stream-like fd 的统一 `noop_llseek` 策略。

## Solution

eventfd 作为 anonymous stream file 实现。VFS inode/path 只提供 fd 身份；readiness、
blocking 和 counter 修改都由 `fs::eventfd` 私有 state owner 负责。eventfd 文件显式使用
`FileMode::STREAM`，因此普通 `read` / `write` 不持有 VFS cursor lock 进入可能阻塞的
后端等待路径。

`O_NONBLOCK` 不在 eventfd state 中缓存。opened file description 继续作为 file status
flags 的唯一真相源，eventfd 后端只通过每次 data I/O 的 `FileIoCtx` snapshot 判断
blocking 与 nonblocking 行为。`F_SETFL` 可以动态改变 `O_NONBLOCK`；`O_APPEND` 对 eventfd
没有 counter 语义影响，但可作为 `F_GETFL` 可见 no-op status bit；`O_DIRECT` 由 eventfd
后端的 status check 拒绝。

blocking read/write 走 latch-style wait：waiter 发布和 predicate 重查在同一把 eventfd
state lock 下完成；状态改变时在锁内 detach 需要唤醒的 read/write/poll triggers，解锁后
再 trigger。poll waiter 保存 trigger 与 interests 只用于唤醒筛选和日志诊断，实际
readiness 每次仍从 counter under lock 重新计算。

## Change

- `anemone-abi` 为 rv64 / la64 新增 `SYS_EVENTFD2 = 19`，并新增 `EFD_*` UAPI 常量。
- 新增 `fs::eventfd` 后端模块：anonymous inode 构造、`FileMode::STREAM` opened file、
  counter state、read/write/poll FileOps、latch wait queues 和 trigger-after-unlock。
- 新增 `fs::api::eventfd2` syscall adapter：校验 flags、创建 eventfd file、初始化
  fd flags/status，并为 eventfd 控制 fd 设置 notification suppression。
- LTP fixture 新增 `CONFIG_EVENTFD=y`，user-test LTP groups 新增 `eventfd` group；当前
  active profile 切到 `eventfd` 便于定向验证。

## Tracking Issues

### CHG-001 - zero-length eventfd I/O

**Status:** Deferred
**Severity:** Euclid

**Issue:** Linux eventfd 对 `count < 8` 的 read/write 返回 `EINVAL`，包括
`count == 0`。当前 Anemone generic read/write helper 在 `count == 0` 时先返回 `0`，
eventfd 后端看不到该请求，因此本轮只覆盖非零 short buffer 的 `EINVAL`。

**Resolution:** 本轮不为单个 eventfd 扩展通用 read/write direct userspace hook 或
`FileMode` 能力。后续如果要补齐精确 scalar/vector eventfd I/O，应新增窄入口，让 eventfd
能在 generic zero-count fast path 前验证请求，同时保持 pipe、fanotify 等既有 stream fd
零长度行为不被静默改变。

### CHG-002 - precise scalar/vector copy semantics

**Status:** Deferred
**Severity:** Safe

**Issue:** Linux scalar eventfd I/O 在 `count > 8` 时只访问前 8 字节，尾部坏地址不影响
结果；`readv/writev` 也按总长度跨 iovec 聚合 8 字节。当前通用 kernel-buffer 路径会先
校验或复制完整用户 buffer，并逐 iovec 分段调用后端。

**Resolution:** 该精确 copy 语义不在第一轮 LTP eventfd 目标内。后续若出现覆盖，应与
zero-length 问题一起通过 eventfd 专用 direct userspace I/O hook 收敛。

## Risk / Follow-up

- `eventfd01..05` 与 `eventfd2_01..03` 是第一轮语义验收目标；`eventfd06` 的 AIO
  completion / overflow 行为仍不属于本轮范围。
- 本轮没有提供内核内部 `eventfd_ctx_*` provider API；AIO completion、fdinfo 和 epoll
  专门行为需要另起小迭代。
- 若后续要统一 socket/FIFO/eventfd readiness 或 stream fd seek 兼容，应升级为更大的
  FileOps/iomux 边界设计，而不是在 eventfd 单点继续扩大。
