## 2026-07-12 - Sched Split Pending Resched Pick Acknowledgement

**Date:** 2026-07-12
**Authors:** doruche, Codex
**Area:** scheduler core / kernel preempt / pending resched

**Summary:** `sched-wait-preempt-arming` 的 post-close feedback 关闭了 processor pending-resched
lifecycle 缺口。`PendingResched` 现在被定义为面向下一次 owner-CPU full pick 的合并 latch；

仍不清除，destructive take 的 caller 继续在 deferred 后恢复 snapshot。未引入事件队列、epoch 或
task-owned cause history。

`mdbook build docs` 通过；pending-resched clear 通过 owner-boundary source audit 验证，不保留只测试
字段赋值的 wrapper / KUnit。有时限启动在 KUnit 和 init 成功后由主机 timeout 终止，随后 signal LTP

## 2026-07-14 - CPU 逻辑 ID 与物理 ID 分离

**Date:** 2026-07-14
**Authors:** EDGW_, Codex
**Area:** CPU discovery / bootstrap / per-CPU / scheduler / interrupt / architecture

**Summary:** `CpuId` 现在只表示从 0 连续分配的逻辑 CPU ID，`PhysCpuId` 表示
固件或硬件可见 ID。架构 early scan 注册 CPU，BSP 单写、封存后无锁只读的静态 registry
同时作为映射和 CPU 数量的唯一真相源。用户运行暴露 `PhysCpuId(3)` 索引长度 3 的
guarded-stack 表越界后，容量进一步拆为 platform `MAX_PHYS_CPU_ID` 与 kconfig
`MAX_LOGICAL_CPUS`；越界物理 CPU warning 后跳过，逻辑超限时保留 BSP 和前
`MAX_LOGICAL_CPUS - 1` 个 AP。逻辑/物理固定 per-CPU backing 分别由默认容量的 `CpuTable`
和 `PhysCpuTable` 限定索引类型；两种 table 的槽位都内建 `CachePadded<T>`，避免再次由裸数组混用
两种容量或在调用点隐藏缓存布局。

**Validation:** agent 侧 `visionfive2-rv64` 与 `qemu-virt-la64` 的 `just build` 通过，
`git diff --check` 通过；按用户要求未进行 LoongArch 运行时验证。用户确认 RISC-V 实机
启动和运行通过；agent 未持有该次运行的原始日志。容量拆分修正后，agent 只重新完成
`visionfive2-rv64` 构建和静态审计；用户要求停止后续测试，LoongArch 构建与 VisionFive 2
运行时复验由用户接手；用户随后确认 VisionFive 2 运行通过，原 guarded-stack 越界未再出现。
最终统一 cache-padded table 布局与 LoongArch correction build 按用户要求未由 agent 运行；未运行项
