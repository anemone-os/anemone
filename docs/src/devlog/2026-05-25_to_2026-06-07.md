## 2026-05-28 - `close_range` 目录化与 `UNSHARE` 文件表拆分

**Date:** 2026-05-28
**Authors:** doruche, Codex
**Area:** fs / syscall / task / user-test / docs

**Summary:** 落地 `close_range` syscall，并把 `close` 与 `close_range` 统一放进 `fs/api/close/` 目录模块。`close_range` 支持 `CLOSE_RANGE_UNSHARE` 和 `CLOSE_RANGE_CLOEXEC` 两个 Linux flag，`first > last` 返回 `EINVAL`，`close_range02` 在 rv64 `fs-fd` 中通过。

**Motivation / Symptom:** 这次的重点不是单纯补一个批量关闭接口，而是把 `CLOSE_RANGE_UNSHARE` 语义补完整。`close_range` 如果只在现有共享 fd 表上批量改状态，会把通过 `CLONE_FILES` 共享同一张表的其他 task 一起影响掉，和 Linux 的“先 unshare 再操作”语义不一致。

**Change:** 将 `close` 拆成目录模块：`close/mod.rs` 只负责模块组织，`close/close.rs` 放单 fd close，`close/close_range.rs` 放 `CloseRangeFlags` 和 `sys_close_range`。`Task::files_state` 改成“当前 task 持有一个可替换的文件表句柄，句柄再指向共享的 `FilesState`”，这样 `UNSHARE` 可以让当前 task 替换成 `fork()` 出来的新表，而不会改动其他共享者；`CLOEXEC` 则只批量设置 `FD_CLOEXEC`。`clone(CLONE_FILES)` 路径也同步改成共享这个句柄。

**Validation:** `just build` 通过；`scripts/run-user-test-rv64.sh rootfsconfig-rv etc/sdcard-rv.img build/user-test-fs-fd.log` 中 `close_range02` 通过，`close_range01` 失败点是测例环境拿不到 device，属于 infra 问题。

**Follow-up:** 如果后续还要补其他 fd 批量操作，可以继续沿这个目录结构拆分；当前只关心 `close_range` 的语义闭合。

**Related:** [活动登记册](../register.md), [当前限制](../register/current-limitations.md)

## 2026-05-27 - 进程组与会话 stage-1 主干

**Date:** 2026-05-27
**Authors:** doruche, Codex
**Area:** task topology / process group / session / signal / wait

**Summary:** 以 `69bff4b` 作为尚未引入进程组和会话的代码基线，在其后建立 stage-1 进程组/会话主干：`TaskTopology` 增加 session 与 process-group authority，`ThreadGroup` 缓存 `pgid` / `sid` / `has_executed`，并接入 `setpgid`、`getpgid`、`setsid`、`getsid`、process-group `kill` 和 `wait4` 的进程组选择语义。

**Motivation / Symptom:** 在 `69bff4b` 之前，任务拓扑只有 task / thread-group / parent-child 关系，无法表达 POSIX process group、session、fork/exec 竞争窗口，也无法让 shell、BusyBox 或后续 job-control 相关测例按 PGID/SID 观察进程。`kill(-1, sig)` 另有一个锁序遗留问题，需要在广播目标快照后再递送信号。

**Change:** 引入 `ProcessGroup` 和 `Session` 一等对象，保持 `TaskTopology` 为唯一 membership authority；fork 继承父进程的 PGID/SID，exec 只标记 `has_executed`，exit 在最后一个 thread 离开时清理 process-group/session membership。syscall 层实现 stage-1 `setpgid` / `getpgid` / `setsid` / `getsid`，`kill(0, sig)` 与 `kill(-pgid, sig)` 走 process-group 广播，`wait4(0, ...)` 与 `wait4(-pgid, ...)` 按 child `ThreadGroup::pgid()` 选择。`kill(-1, sig)` 后续改为先快照除 init 和调用者自身外的 thread group，再释放 topology 锁逐个递送，避免在 topology 锁内进入 signal 递送路径。

**Validation:** 针对该语义新增 `pg-test` 用户态覆盖；2026-05-27 的 `kill(-1, sig)` 补齐后，`just build` 通过。本文档更新时未重新跑完整 rv64 `user-test` 链路。

**Follow-up:** 继续补齐 `waitid`，再接 controlling tty、foreground/background process group、orphaned process group 的 `SIGHUP` / `SIGCONT` 规则，以及完整 shell job-control 行为。更广泛合入前仍需要 rv64 端到端回归。

**Date:** 2026-05-27
**Authors:** doruche, Codex
**Area:** fs / syscall / vectored IO / user-test

**Summary:** 针对新加入的 write 相关 LTP 测例，修正 `pwrite04` / `pwrite04_64` 暴露的 `O_APPEND + pwrite` 语义，并补上 `pwritev2` 的 stage-1 syscall 路径，使 `pwritev201` / `pwritev202` 及 64 位变体回到预期 errno 和偏移行为。

**Motivation / Symptom:** `pwrite04` 系列把带 `O_APPEND` 的 fd 传给 `pwrite`，Linux 兼容语义是写入仍追加到 EOF，但 positioned write 本身不推进当前 file offset；之前实现把这条路径拒成 `EINVAL`。`pwritev202` 系列则同时覆盖 `pwritev2` 的非法 offset、非法 flags、坏 iovec、坏 fd 和 pipe `ESPIPE` 等错误路径，当前内核缺 syscall 入口和 `offset == -1` 的 current-position 语义。

**Follow-up:** `pwritev2` 的 `RWF_*` flags 仍只是明确拒绝，后续如果要支持 `RWF_APPEND`、per-IO sync、nowait / hipri 等语义，需要单独扩展并回归。`write05` 的 `SIGPIPE` 问题已登记为开放问题。

## 2026-05-28 - `CLONE_VFORK` 主路径落地

**Date:** 2026-05-28
**Authors:** doruche, Codex
**Area:** task / clone / execve / exit / docs

**Summary:** 完成 `CLONE_VFORK` 的主路径接线：在 `Task` 上加入 child-owned 的 `vfork_done` 事件，`clone(CLONE_VFORK)` 在 child 发布后等待该事件，`execve` 与 `exit` 分别在完成点唤醒父进程。这样 vfork 的父阻塞与 child 退出/切换到新映像的解除条件收敛到同一条事件路径上。

**Motivation / Symptom:** 之前内核只有普通 `clone` / `fork` 的返回路径，没有把 `CLONE_VFORK` 的“父进程必须等待 child 先完成 `execve` 或退出”这条语义单独闭合起来。用户态传入 `CLONE_VFORK` 后，父子调度关系仍会回到普通 clone 的默认行为。

**Change:** `Task` 增加 `vfork_done: Event` 字段并在创建路径中初始化；`clone` 的 flag 白名单加入 `VFORK`，在发布 child 后对其 `vfork_done` 做 uninterruptible wait；`execve` 在切换到新映像并恢复到用户态前 publish `vfork_done`，`exit` 在发布 `child_exited` 之后也 publish 一次，保证 child 无论是执行新映像还是直接退出都能解除父等待。

**Validation:** `just build` 通过。

**Follow-up:** 后续如果要补 `vfork` 的更细分边角行为，再单独看 `CLONE_PARENT` / `CLONE_THREAD` 等组合，以及用户态测例覆盖。

## 2026-05-28 - fs 组 `getcwd` 语义收口

**Date:** 2026-05-28
**Authors:** doruche, Codex
**Area:** fs / syscall / ramfs / procfs / user-test

**Summary:** 针对新加入的 fs 组 `getcwd` LTP 用例，修掉了三个直接语义问题：坏用户指针现在按 `EFAULT` 返回而不会内核态写空指针 panic；`getcwd` 返回值恢复为相对进程 root 的绝对路径；`openat` 在非 create 路径不再校验被 libc 传入的无意义 `mode` 参数。同时给 `ramfs` 补了 regular-file rename 的 stage-1 实现，使 `getcwd04` 不再卡在 `EOPNOTSUPP`。

**Motivation / Symptom:** `getcwd01` 最初在无效用户缓冲区上没有稳定返回 `EFAULT`，而是先报错码不对，随后触发内核页错误。完整 fs 组日志还暴露了 `getcwd02` 期望 `/tmp` 却拿到错误路径、`musl getcwd03` 因返回路径不是绝对路径而失败、`getcwd04` 因 `rename` 未实现而 `TBROK`。`build/ltp-debug.log` 进一步显示 musl `realpath` 会在非 `O_CREAT` 的 `openat` 上传入垃圾 `mode`，Linux 语义应忽略该参数。

**Change:** 用户地址校验先拒绝越过 `USPACE_TOP_ADDR` 的地址并把非法地址统一映射到 `BadAddress`；`Task::rel_cwd()` 对非根 cwd 返回 `"/" + rel`，避免把 chroot/root-relative 路径变成相对路径；`openat` 的 syscall 入口把 `mode` 当作原始 `u32`，只在 `O_CREAT` / `O_TMPFILE` 分支解析权限位。`ramfs` 的 `rename` 现在支持同一 superblock 内非目录项改名和覆盖普通目标，支持 `RENAME_NOREPLACE`，目录 rename 仍明确拒绝。

**Validation:** `just build` 通过；复跑 `scripts/run-user-test-rv64.sh rootfsconfig-rv etc/sdcard-rv.img build/user-test-rv64.log` 后 QEMU 正常关机。结果为 `attempted=12 passed=8 failed=4`：glibc 侧 `getcwd01`、`getcwd02`、`getcwd03`、`getcwd04` 全部通过；musl 侧 `getcwd01`、`getcwd03`、`chdir04` 通过。剩余失败中，`chdir01` 仍是已登记的设备池 infra 问题，`musl getcwd04` 是测例要求至少 2 个 online CPU 的条件问题，`musl getcwd02` 已从旧的 `openat EINVAL` 前进到 `realpath()` 依赖 `/proc/self/fd/<n>` 时返回 `ENOENT`。

**Follow-up:** `/proc/<tgid>/fd` 不做针对 `getcwd02` 的临时补丁，后续等 procfs/fd 目录、fd 生命周期和 fd symlink readlink/open/stat/readdir 语义一起作为系统性框架引入。`ramfs` 目前只补 regular-file rename，完整 directory rename 也单独登记为后续限制。

## 2026-05-28 - ROFS 直接写路径与 `mkdirat02` 收口

**Date:** 2026-05-28
**Authors:** doruche, Codex
**Area:** fs / mount / VFS / syscall / user-test

**Summary:** 针对 fs 组新增的 `mkdirat02`，补齐 `mount(MS_RDONLY)` 到 `MountFlags::RDONLY` 的传播，并把只读挂载上的直接写操作统一映射到 `ReadOnlyFs -> EROFS`。最初只在 `vfs_mkdir_at` 上补了检查以解除 `mkdirat02` 的 TFAIL，随后把同类直接写入口扩展到 VFS 目录项修改、普通文件 open-for-write / write / truncate、`chmod` / `chown` / `utimensat` / `fallocate`。

**Motivation / Symptom:** `build/user-test-rv64.log` 中 `mkdirat02` 的两个 EROFS 分支失败：在只读 `mntpoint` 下创建新目录时意外成功；在只读挂载下目标已存在时返回 `EEXIST`，而 LTP 期望 `EROFS` 优先。日志同时显示测试框架已经把 `mntpoint` 按只读重新挂载，但内核此前完全忽略 mount flags，因此 VFS 创建路径看不到 ROFS 状态。

**Change:** ABI 层加入 Linux `MS_RDONLY` 常量；`sys_mount` 解析该 flag 并保存为 `MountFlags::RDONLY`，其他 mount flags 继续作为 unsupported flags 打日志后忽略。VFS 层新增 `Mount::ensure_writable()` 并在 `touch`、`mkdir`、`link`、`symlink`、`unlink`、`rename`、`rmdir` 前检查目标 mount；普通文件内容写入、open-for-write、truncate/ftruncate、fallocate、chmod/chown、utimensat 也在直接修改前检查只读挂载。

**Validation:** `just build` 通过；复跑 QEMU 后 `build/user-test-rv64-mkdirat02-fix.log` 中 glibc/musl 的 `mkdirat02` 均通过，两个 EROFS 分支返回 `ReadOnlyFs -> EROFS (30)`，两个 ELOOP 分支仍通过。fs 组仍有其他与本次 ROFS 无关的失败，例如 `mkdir03` / `mkdir04` 受缺失 syscall 或 credentials 路径影响。

**Follow-up:** 当前 ROFS 仍是 stage-1 的直接写路径防护，不承诺 shared writable mmap、writeback、remount/bind/move mount 或完整 Linux mount flag 矩阵；该限制已登记到 current limitations。
