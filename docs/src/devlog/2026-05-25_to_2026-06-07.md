## 2026-05-26 - signal restorer 兼容 glibc 2.3.5 / loongarch 观察

**Date:** 2026-05-26
**Authors:** doruche, GitHub Copilot
**Area:** signal / ABI / user-test / docs

**Summary:** 确认当前用户态测例的信号 ABI 仍以 legacy `sa_restorer` 路径为准，主目标是兼容 `musl 1.2.2` 的 `pthread_cancel` 测例；`glibc 2.3.5` 仅作为 `sigaction` / `sa_restorer` 形状对照参考。同时暂不引入面向新内核头的条件编译分支，避免为了新头文件语义切换破坏现有回归。

**Motivation / Symptom:** 现有失败点首先出现在 `musl 1.2.2` 的 `pthread_cancel` 测例；随后再拿 `glibc 2.3.5` 的 `sigaction` 结构和 `sa_restorer` 语义做交叉对照。现代 LoongArch / asm-generic 头文件不再鼓励新架构继续定义 `SA_RESTORER`，但当前阶段更重要的是保住既有 libc-test 兼容面，而不是把用户态 ABI 切到一套新的条件分支。

**Change:** 保持内核侧对 restorer 路径的兼容处理，继续按测例需要提供 signal return trampoline；文档层面明确本阶段不引入针对新内核头的条件编译兼容层，避免把 ABI 选择复杂化。

**Validation:** `musl 1.2.2 pthread_cancel` 测例通过；用户确认完整 rv64 测试链路可跑通。

**Follow-up:** 如果后续要同时覆盖新旧内核头的用户态 ABI，需要单独设计可配置的 signal UAPI 适配层，并补足 glibc / musl 两侧的回归矩阵。
