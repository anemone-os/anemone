## 2026-05-22 - glibc libc-test 与 stage-1 O_TMPFILE

**Date:** 2026-05-22
**Authors:** doruche, GitHub Copilot
**Area:** VFS / openat / glibc libc-test

**Summary:** 为 openat 增加 stage-1 O_TMPFILE create-open-unlink 实现，修复 O_TMPFILE 复合 O_DIRECTORY 位导致的 NotDir 误判，并把剩余 glibc libc-test 的 fscanf 与 fwscanf 失败归类为宿主可复现的 glibc 语义差异。

**Motivation / Symptom:** glibc libc-test 静态运行中，tmpfile 路径最初会在 openat 处触发错误；修复后，fscanf 与 fwscanf 仍然失败，需要继续区分这是内核回归、rootfs 环境缺口，还是 glibc 与该测试集预期不一致。

**Change:** 在 anemone-kernel/src/fs/api/openat.rs 中实现了 stage-1 O_TMPFILE 路径，并补了内核态 kunit。随后修正了 O_TMPFILE 常量复合 O_DIRECTORY 位后被通用目录检查误判为 NotDir 的问题。对 glibc libc-test 相关失败重新分类：clocale_mbfuncs、fnmatch、fscanf、fwscanf 在主机 qemu-riscv64 glibc 上可直接复现；mbc 依赖可用的 UTF-8 locale 数据，因此不再把这些项直接视为内核问题。

**Validation:** just build；qemu-riscv64 etc/rootfs-rv/glibc/entry-static.exe fscanf；qemu-riscv64 etc/rootfs-rv/glibc/entry-static.exe fwscanf；内核日志中不再出现 openat failed with error NotDir。

**Follow-up:** 为 glibc 维护 expected-fail 或 skip 基线，避免把宿主可复现的 glibc 差异误报成内核回归；如果后续需要完整 O_TMPFILE 语义，还需要匿名 inode、linkat 与 AT_EMPTY_PATH 支持。

## 2026-05-23 - truncate/ftruncate 模块整理与 KUnit 去 task 依赖

**Date:** 2026-05-23
**Authors:** doruche, GitHub Copilot
**Area:** VFS / truncate / ftruncate / docs

**Summary:** 按 fs/api 现有组织方式把 truncate 与 ftruncate 拆到独立文件，并把 truncate 相关 KUnit 改成只验证 inode 级核心逻辑，避开 bspinit hanging FsState 带来的路径解析 panic。

**Motivation / Symptom:** 之前把 truncate 与 ftruncate 混放在单文件里，不符合当前 syscall 模块的目录习惯；同时 KUnit 通过 current task 做路径解析和 fd 打开时，会落到 bspinit kthread 的 hanging FsState，导致测试路径直接 panic。

**Change:** 将 anemone-kernel/src/fs/api/truncate.rs 重组为 anemone-kernel/src/fs/api/truncate/mod.rs、anemone-kernel/src/fs/api/truncate/truncate.rs 和 anemone-kernel/src/fs/api/truncate/ftruncate.rs；移除了不必要的长度 helper，在 syscall 内直接展开负长度检查；把 KUnit 改为使用 vfs_touch、vfs_open 和 inode.truncate 覆盖 shrink、extend 后零填充以及目录拒绝语义，不再测试依赖 task fsstate 的 syscall 包装层。

**Validation:** VS Code Rust 诊断通过；just build。

**Follow-up:** 如果后续要收敛 truncate 与 mmap 的可见性语义，需要补活跃文件映射的失效或回收路径，而不只是更新 inode 大小与驻留页缓存。

## 2026-05-23 - ext4 truncate 页缓存整理与原因登记

**Date:** 2026-05-23
**Authors:** doruche, GitHub Copilot
**Area:** ext4 / truncate / page cache / docs

**Summary:** 整理了 ext4 truncate 的内存侧控制点，把 post-truncate 的 in-memory 处理从 mapping 层收回到 inode 私有状态，并进一步把全量页缓存失效收敛到“仅失效可见性发生变化的页”。

**Motivation / Symptom:** 在 ext4 上执行 shrink 到 2 字节再 grow 到 5 字节时，磁盘侧 set_len 与 zero-fill 已经成功，但后续通过 inode.mapping().read() 和普通文件读仍可能看到旧的 cde 尾字节。这说明故障不在 reopen 路径，而在 ext4 resident page cache 对 partial-page truncate 的内存侧可见性。

**Change:** 将 ext4 grow 场景的零填充逻辑抽成 inode truncate 路径中的独立 helper，把 in-memory 的 truncate 收尾改成 Ext4RegState 级别的 apply_truncate，并删除了几乎没有语义价值的 Ext4RegMapping::truncate 包装。根因定位收敛到 boundary page：只要 resident page 覆盖了 truncate 前后“可见字节范围变化”的那一页，就不能继续信任它的内存内容。磁盘侧 set_len 与 grow zero-fill 已经是权威状态，因此现在只失效受影响页，再由后续 load_page 按需回到磁盘内容；未受影响的热页则继续保留。

**Validation:** just build；用户在文件系统重构后重新运行 truncate KUnit，确认 ext4 shrink-then-extend 零填充测试通过。

**Follow-up:** 这次整理把全量失效收敛到了页粒度，但还没有把“为什么旧 resident boundary page 在原位修补后仍能被后续读取看到旧尾字节”缩到单一代码点。当前更稳妥的判断是 ext4 现有页缓存模型对 partial-page truncate 缺少可靠的一致性边界：覆盖可见性变化范围的 boundary page 必须先失效或重载，不能直接延续旧页内容。后续若要进一步替换掉页粒度失效，需要继续把这个 boundary-page coherence 问题缩到明确的不变量或单点 bug。
