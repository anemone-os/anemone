# 文档框架

## 目标

这本文档不是 API 参考，也不是放长篇设计散文的地方。

它的职责是保存高信号、可操作的知识，让人和 coding agent 能够快速回答五个问题：

1. 这段行为由仓库的哪一部分负责？
2. 当前有哪些东西是坏的、未完成的、或脆弱的？
3. 最近改了什么，为什么改？
4. 当前测例或兼容性状态如何？
5. 哪些设计决策已经足够稳定，后续工作应当尊重？

## 应该维护什么

| 文档对象              | 是否维护             | 为什么值得维护                                  | 何时更新                       |
| --------------------- | -------------------- | ----------------------------------------------- | ------------------------------ |
| 共享开发日志          | 是                   | 让 bug 回溯和上下文恢复的成本很低               | 每次完成一个有意义的任务后     |
| 小迭代记录            | 按需                 | 保存小修、小调查和局部语义变化的问题、方案、验证与局部跟踪 | 双周日志容纳不下但又不需要 RFC 时 |
| 已知问题与当前限制    | 是                   | 让 agent 能区分“新回归”与“已接受的缺口”         | 状态变化时，或重新确认后       |
| Testsuits 通过情况    | 是                   | 让功能覆盖与 bench 可运行性具备可扫描的共享视图 | 测例集启用状态或运行结果变化时 |
| 决策记录              | 是，但只记录重要决策 | 保留原因、约束和被拒绝方案                      | 出现跨子系统决策时             |
| 公开草案 / RFC        | 按需                 | 为未定稿但需要共享评审的方案提供公共载体        | 进入跨人协作评审时             |
| 代码库地图            | 是，但要浅           | 帮助新成员和 agent 快速找到 owning surface      | 所有权边界变化时               |
| 事故复盘或 postmortem | 可选                 | 对高影响问题有长期价值                          | 发生严重回归后                 |

## 不应该维护什么

- 没人会持续更新的大型设计文档。
- 只是重复代码含义的生成式或手写 API 文档。
- 另一套重复 GitHub issue、TODO 文件或聊天记录的任务系统。
- 与仓库自动化 skill 重复的构建或验证操作手册。
- 未公开的个人草稿、私有环境配置和原始调试输出。

活动登记册不是 backlog。GitHub issues / PR 负责工作项、讨论、验收条件和长线程；活动登记册只保留当前仍然生效、会改变诊断或验证路径的操作性事实，并尽量通过链接指向对应 issue / PR，而不是重复复述。

如果一份文档主要是在复述代码里已经一眼能看出的机制，它就会很快腐烂。相比之下，更应该记录不变量、决策、失效模式和验证闭环。

## 当前事实与公开草案的边界

- 当前事实页面记录“现在是怎样”。
- 公开草案页面记录“可能如何改变”。

共享但未定稿的内容，只有在确实需要多人协作评审时才进入公共仓库。此类页面应统一放在 `docs/src/rfcs/`。大型重构默认使用 `docs/src/rfcs/<short-slug>/` 目录形式，入口为 `index.md`，实施计划为 `implementation.md`，必要时拆出 `invariants.md` 和 `backgrounds/` 历史材料。

RFC 入口应在页首明确给出以下字段：

- `状态`
- `负责人`
- `最后更新`
- `领域`
- `开放问题`
- `下一步`

草案一旦被接受，其结论应转写进当前事实页面、决策记录，或在 RFC 目录内明确标记为 canonical implementation source。草案本身不陈述已经落地的当前实现事实。

## 两位开发者的协作模型

即使只有两位开发者，规范性的历史也应该共享，而不是按人拆分。

- 维护一条共享的开发日志流。
- 每条记录都带 `Authors` 字段。
- 如果同一个任务由两人共同推进，优先写一条联合记录，而不是两条平行个人记录。
- 如果某个问题明确由一人继续推进，在问题条目里使用 `Owner` 字段。
- Testsuits 与活动登记册也应维持共享视图，而不是按人拆分。

最轻的稳定分工通常是：GitHub issues / PR 管工作项与讨论，开发日志管历史，活动登记册管当前活动事实。只要这三者边界不混，文档负担就不会失控。

对小型迭代，双周开发日志负责时间线摘要；`docs/src/devlog/changes/` 下的小迭代记录负责具体问题、局部方案、验证、风险和本迭代内部 tracking issues；活动登记册只保留当前仍然生效的问题或限制。小迭代记录不替代 RFC，也不承载未接受的中大型计划；如果局部 tracker 开始需要仓库级 contract、不变量、阶段 gate 或多轮 review，就应自然升级为 RFC。

这比维护两套个人日志简单，也更利于按时间回放某个子系统的演化。

## 文档质量门槛

这本文档里的长期页面都应该满足以下要求：

- 它写的是那些难以仅靠读代码快速恢复的信息；
- 在新鲜度重要时，它带有日期或最后确认时间；
- 它足够短，维护者愿意持续更新；
- 它通过链接串起日志、问题和决策，而不是重复拷贝内容。

如果某一页已经没人使用，就删除它，或把其中仍有价值的部分归档进开发日志。过时但看似确定的文档，比没有文档更糟。
