# 决策记录

只有当一个决策的重要性足以让未来维护者反复追问“为什么当时这么做”时，才值得写一条简短的 ADR 风格记录。

## 什么时候应该写

当至少满足以下任一条件时，创建一条决策记录：

- 它影响多个子系统；
- 它改变 ABI 或兼容性预期；
- 它引入了后续修改必须遵守的不变量；
- 存在多个看起来都合理的备选方案，而权衡并不能从代码中直接看出来。

日常局部重构不需要写决策记录。

## 记录什么

优先记录稳定的原因和约束，而不是冗长的机制讲解。

每条记录都应回答：

- 当时面对的问题是什么；
- 最终做了什么决策；
- 哪些备选方案被拒绝了；
- 该决策带来了哪些后果与要求；
- 未来出现什么证据时，这个决策应被推翻。

## 存放约定

- 决策记录放在 `docs/src/decisions/` 下。
- 文件名使用日期加 slug，例如 `adr-20260522-signal-delivery-boundary.md`。
- 带一个简短状态标记，例如 `Proposed`、`Accepted` 或 `Superseded`。

## 维护规则

如果后续反转了一个决策，不要把旧文档改写成从未发生。应把它标成 superseded，并链接到新的记录。

ADR 保存“为什么这样决定”的历史，不承担跨 RFC 当前不变量。若决策引入或改变会被多个 RFC / 模块依赖的规则，应链接到 [当前契约](./contracts.md) 中对应的稳定 contract ID；后续变化更新 current contract 和来源记录，不要求读者从多份 ADR 回放当前规则。
