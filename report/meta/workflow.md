# 报告写作流程

## 入口顺序

写某个章节前，按顺序阅读：

1. `report/requirements.md`
2. `report/meta/style.md`
3. 本文件
4. `report/meta/outline.md`
5. `report/meta/sources.md`
6. 目标 `report/content/*.typ` 文件

如果任务涉及 Typst 版式，还要阅读 `report/main.typ`、`report/conf.typ` 和
`report/components/*.typ`。

## 证据门槛

每个技术小节在写正文前，至少需要一个证据入口。证据入口用于核对事实、避免编造；
它不要求正文直接展示具体路径、函数名、字段名或源码调用链。证据入口可以是：

- 代码路径、类型、函数、模块或 syscall 实现。
- 测例、本地测试脚本、LTP case、profile 或端到端日志。
- RFC、transaction devlog、小迭代 devlog、register 记录或 current limitation。
- 明确的问题、失败现象、死锁、竞态、errno 不匹配、panic 或性能问题，并且有足够
  上下文可以核对。

如果缺少证据：

- 不得编造事实。
- 不得用泛泛而谈的文字填充小节。
- 留下 TODO 或简短章节备注，说明缺少什么来源。

正文呈现时，默认把证据转写成机制和策略说明。除核心结构体、关键代码片段、代表性
syscall/ABI 名称、公开接口名或必须精确定位的调试案例外，不要在正文中堆代码标识符。
评委需要先理解系统做了什么、为什么这样做、行为边界是什么；源码定位可以留给附录
备注或现场需要时的口头说明。

## 粒度控制

起草时先选中心信息，再决定是否展开细节。一个模块小节通常只需要讲清几个关键点：
owner 边界、核心机制、代表性路径、验证或重要取舍。除非细节直接支撑这些点，不要把
状态机、字段、函数和调用顺序完整拆开讲。

压缩草稿时，不要优先删掉未来路线或项目价值；先删掉源码导览、OS 常识、重复机制和
无法服务评审判断的细节。评委每天会阅读大量报告，正文应让他们快速看到 Anemone 的关键
设计和工程判断，而不是要求他们跟着源码走一遍。

## 限制与未来工作

当前限制不是段落模板，也不是 agent 可以自行补充的防守性说明。只有当用户大纲、章节
安排或明确指令要求写限制 / 边界时，才写入正文；否则禁止主动加入“当前限制”“当前
边界”“阶段性边界”“仍未支持”等表述。确实发现值得提醒的缺口时，放在工作备注或回复
中请用户决定，不要直接塞进正文。

当大纲要求写限制时，只覆盖大纲指定的范围；不要顺手补相邻缺口，不要把未来路线改写
成限制清单。

大纲中的未来工作、项目价值和竞赛表达默认保留，可以出现在摘要、总结、展望或相关
模块末尾。处理这类表述时，不要让 agent 自行判断“宣传”“空泛”或“不可验证”后删除。
只有明确事实冲突、与用户大纲冲突，或用户明确要求清理时才改；否则保留用户的正向
表达，并把正文组织成成果、机制和路线说明。
如果 agent 认为大纲里的正向表达存在竞赛表达风险，但没有明确事实冲突，应暂停该处
写作并向用户确认“保留 / 降调 / 删除”，不能自行决定。

未来工作必须是报告内容，不是写作指令。禁止在正文中保留“如果这里要扩展，未来可以
怎么写 / 怎么讲 / 怎么补证据”这类元叙述。遇到这种句子时：

- 如果它只是写作提示，直接删除或移入 TODO。
- 如果它包含真实路线，改写成直接的项目展望。
- 如果它需要证据但当前没有证据，留下章节备注，不要伪装成正文。

## 草稿形状

模块小节优先按这个顺序写：

1. 这个模块负责什么。
2. 主要机制、数据关系或生命周期是什么。
3. 一个具体机制或代表路径。
4. 有证据时，写一个代表性问题或取舍。
5. 大纲要求时，写验证状态、当前边界或后续方向。

工程过程和 AI 小节优先按这个顺序写：

1. 规则或原则。
2. 具体例子。
3. 边界或失败模式。
4. 验证或纠正方法。

## 与其他材料的关系

`anemone-book/` 是材料来源，不是报告风格来源。只能在核对代码、RFC、devlog、
register 或 current limitations 之后，复用其中的事实、模块地图和候选例子。

`report/refs/*.pdf` 是风格和结构参考。不得复制其他项目的具体说法、图或结论。

`report/requirements.md` 是评审要求来源。如果风格选择和评审要求冲突，以评审
要求为准。

## 审查清单

完成一段报告正文前，检查：

- 每个技术小节是否有证据入口？
- 主语是否是具体 OS 机制，而不是抽象口号？
- 是否只讲了关键机制和代表性细节，而不是源码级穷举？
- 正文是否把证据转写成机制 / 策略说明，而不是源码复述？
- 除核心结构体、关键代码片段或必要定位点外，是否避免了具体路径、函数名和字段名？
- 完成情况、性能、排名、测试结果是否可验证？
- 是否没有在大纲之外自行加入“当前限制”“当前边界”或类似防守性表述？
- 是否没有因主观判断“宣传”“空泛”“不可验证”而删除用户大纲里的正向表达？
- 对拿不准的正向表述，是否停下来询问用户，而不是自行裁剪？
- 正文是否删除了“如果这里要扩展”“正式正文需要”等元叙述？
- AI 使用说明是否包含工具、范围、人工审查和验证？
- 这段话是否像队员提交给评委的成果报告？

## 验证

修改 Typst 或正文后运行：

```sh
make -C report
git diff --check -- report
```

新增未跟踪文本文件时运行：

```sh
git diff --no-index --check -- /dev/null <file>
```

大幅修改后，用 `pdftotext` 检查摘要、目录、表格和明显版式回归。
