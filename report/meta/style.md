# 报告写作风格

## 写作声音

报告的声音是：队员亲自整理的工程说明和成果陈述。

每段技术正文优先回答：

- 我们做了什么？
- 为什么需要做？
- 它在系统机制中处于什么位置？
- 遇到了什么问题？
- 如何验证？
- 如果大纲要求，边界、限制或下一步是什么？

允许使用“我们”，尤其是在说明取舍、调试、验证、团队过程、AI 使用和限制时。
主体技术机制小节优先让机制本身作主语；只有讲决策、问题、取舍或验证时再用
“我们”。

## 代码引用边界

报告正文不是源码导览。写作前可以用代码、日志和文档核对事实，但正文默认只解释
机制、策略、语义边界和工程取舍。

只有以下情况适合在正文中直接出现具体代码标识符、文件路径或代码片段：

- 核心结构体或核心数据结构，读者必须看到其形状才能理解设计；
- 代表性 syscall、ABI 名称、公开接口名或评审需要识别的关键路径；
- 一小段能说明机制的代码片段，且文字无法更清楚表达；
- 具体 bug / 调试案例中必须说明定位点。

其他情况下，不要罗列函数名、字段名、文件路径或调用顺序。应该把源码事实转写成
“创建时如何继承资源”“exec 边界保留和替换什么”“wait 如何区分退出和回收”这类
机制说明。代码路径、字段和函数名可以作为起草时的证据备注，不应成为正文主体。

## 机制展开粒度

报告不是内核教材，也不是 RFC 证明。评委通常已经熟悉 fork / exec / wait、页表、
VFS、pipe、timer、中断和 syscall dispatch 这些基本概念；正文不需要逐状态、逐字段、
逐分支复述机制。

一个技术小节优先保留 2-4 个关键点：

- 这个能力域在 Anemone 中由谁负责。
- 和普通教材实现相比，最值得说明的结构、边界或取舍是什么。
- 一个能支撑理解的代表性路径、问题或测例。
- 大纲要求时，说明验证结果、限制或下一步。

只有当细节能体现项目区分度、解释复杂 bug、说明 ABI 边界，或避免
误解时，才展开到具体状态和调用顺序。常规控制流、显而易见的对象关系和读者能自然
推断的步骤应压缩成一句机制说明。

如果草稿已经把机制拆得过细，修改时先保留中心判断，再删掉以下内容：

- 只是在解释 OS 常识的段落。
- 没有服务于取舍、验证或评审判断的状态枚举。
- 连续罗列函数、字段、文件和调用顺序的源码导览。
- 重复说明前文已经建立的模块职责。

## 限制与展望口径

限制和边界不是 agent 自行补充的“诚实说明”。除非用户大纲、章节安排或明确指令已经
要求写限制 / 边界，否则正文不要主动加入“当前限制”“阶段性边界”“仍未支持”等防守性
表述。即使来源材料中能查到缺口，也先放在工作备注或回复中请用户决定，不要替用户把
报告改成缺陷清单。

当大纲确实要求写限制时，只写与该大纲项直接相关的限制，并保持简短、具体、可验证。
不要顺手扩展到相邻缺口，不要为了显得谨慎而追加新的当前边界。

大纲中的后续策略、开源价值、工程愿景和竞赛表达默认保留。不要仅仅因为它们面向未来
就改写成当前限制或直接删除；正文应围绕它们组织已有成果、关键机制和项目路线。

不要让 agent 自行判断“这是不是宣传口吻”“这是不是空泛”“这是不是不可验证承诺”。
这些判断太主观，不能作为删除或降调的理由。处理正向表述时只看三件事：

- 是否与已核对事实直接冲突。
- 是否与用户大纲直接冲突。
- 用户是否明确要求清理这类表达。

三者都不满足时，保留用户大纲里的正向表达，并围绕它补机制、事实或结构；不要主动
改成当前限制、当前边界或缺陷披露。

如果确实担心某句正向表达会显得过强，但无法证明事实冲突，停下来问用户：保留、降调
还是删除。不要把这个判断藏在润色里自行处理。

## 禁止元叙述

报告正文不得保留写作过程、章节规划、作者提醒或 agent 操作提示。尤其不要写：

- `如果这里要扩展，未来可以继续说明...`
- `正式正文需要补齐...`
- `本节可以进一步展开...`
- `这一段应当结合某某证据...`

这类话可以存在于 meta 文档、草稿备注或 TODO 中，但不能进入提交给评委的正文。
正文里的未来工作必须是项目路线本身，而不是“这段文字以后怎么写”或“这个机制以后
怎么被讲清楚”。如果要保留未来方向，直接写能力演进：现有基础是什么、要补哪个缺口、
服务哪个能力域；不要写条件式扩展说明。

## 去 AI 味规则

避免反复使用 `Anemone 不是...而是...` 这类辩论式开场。优先直接说明机制。

避免机械模板句：

- 不要写：`本系统通过完善的同步机制显著增强了进程间通信能力。`
- 可以写：`我们在 pipe 关闭路径中唤醒等待队列，避免读写端退出后等待者长期阻塞。`

避免 AI 总结式段落，例如结尾写 `提升了系统的可维护性和扩展性`。如果这个判断
重要，就写清楚具体机制、调试收益或后续演进路线。不要把正向展望当作宣传稿删除；
要改的是模板腔，不是用户想表达的成果、优势和竞赛表达。

## 抽象词边界

报告可以使用 `anemone-book` 中的词，例如 `owner boundary`、`single source of
truth`、`ABI surface`、`native contract`，但它们只能作为局部解释工具。

不要让这些词成为章节结构主语。章节主语必须是 OS 能力域和具体机制：系统调用
处理、任务调度、虚拟内存、VFS、文件描述符、pipe、procfs、devfs、trap、中断、
上下文切换、双架构适配、测试和调试。

## 开发过程写法

不要把整份报告写成 devlog。主体章节应以模块设计和实现说明为主。

每个核心章节可以放 1-2 个代表性问题片段，但只有当它能解释设计选择时才放。
问题片段按这个形状写：

1. 问题来源：测例、日志、代码审查、真实负载或失败现象。
2. 定位过程：检查了什么源码或行为。
3. 实现落点：改动了哪个模块或机制；只有必要时才写具体路径。
4. 验证结果或剩余限制。

更完整的开发过程、调试方法、RFC/devlog 工作流和 AI 使用集中放在附录。正文
只保留内核设计层面的章节。

整份报告的大致比例：

- 约 70%：模块设计和实现说明。
- 约 20%：代表性问题、调试、验证和工程过程。
- 约 10%：总结、AI 边界、经验、附录和未来工作。

## AI 使用写法

AI 使用章节必须具体、可追责，至少包括：

- 使用了哪些工具。
- 使用场景。
- 生成内容范围。
- 人工修改和审查方式。
- 验证方法。
- 1-2 个幻觉或错误方向案例，以及如何纠正。

RFC/devlog/register/write-set/review/validation 这套工作流应放在附录中，定位为
约束 AI 和团队开发的工程控制面。不要把它写成宣传式的方法论成果。
