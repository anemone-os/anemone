# 开发报告写作协议

本目录是 Anemone 面向比赛评委 / 老师的开发报告。它不是技术读物，不是 RFC
压缩版，也不是宣传页。

除代码标识符、路径、命令、必要英文技术术语外，报告正文和本目录内的报告
流程文档默认使用中文。

修改报告正文前，必须先阅读：

1. `report/requirements.md`
2. `report/meta/style.md`
3. `report/meta/workflow.md`
4. `report/meta/outline.md`
5. `report/meta/sources.md`

然后再阅读目标章节 `report/content/` 下的 `.typ` 文件。

硬约束：

- 写作口吻应像队员亲自整理的工程说明和成果陈述。
- 报告面向评委 / 老师，不是源码导览或机制证明。默认只讲关键机制、关键取舍、
  代表性路径和可体现 Anemone 区分度的点；评委通常已经知道 OS 基本机制，不要
  把每个状态、字段、分支和调用顺序拆到事无巨细。
- 每个技术小节写作前必须有具体证据入口：代码、测试、日志、RFC/devlog/register
  记录，或明确的问题 / bug。但证据入口是写作前的核对依据，不等于正文必须列出
  具体文件路径、函数名、字段名或调用链。
- 正文默认解释机制、策略、语义边界和工程取舍。除非它是核心结构体、关键代码片段、
  代表性 syscall/ABI 名称，或评审判断必须精确定位的实现点，不要在正文中堆具体代码
  标识符、文件路径和源码复述。
- 没有证据入口的技术判断不得进入正文，只能留下 TODO 或章节备注；有证据的判断也要
  先转写成面向评委 / 老师可读的机制说明。
- 禁止自行引入“当前限制”“当前边界”“阶段性边界”或类似防守性表述。只有当用户大纲、
  章节安排或明确指令已经要求写限制 / 边界时，才可以写；否则即使 agent 认为某个缺口
  真实存在，也不得主动塞进正文。需要提醒用户的事实放在回复或章节备注中，不伪装成
  报告正文。
- 禁止自行裁剪用户大纲里的正向表述。不要因为 agent 自己觉得“宣传”“空泛”“不可验证”
  或“承诺过强”就删除、降调或改写。只有当表述与已核对事实直接冲突、与用户大纲直接
  冲突，或用户明确要求清理该类表达时，才可以改；否则保留，并围绕它组织证据和机制
  说明。
- 如果大纲中的正向表述看起来可能有风险，但没有明确事实冲突，agent 应停止并询问用户
  是否保留、改写或删除；不得自作主张先删掉。
- 坚决禁止写作元叙述。正文不要出现“如果这里要扩展，未来可以 / 应当……”“正式正文
  需要补齐……”“本节可以进一步说明……”这类给作者或 agent 的操作提示。未来工作应
  写成项目路线或能力演进，而不是写作计划、扩展教程或待办说明。
- 主体章节按 OS 能力域和具体机制组织。`anemone-book` 中的抽象设计词可以用来
  解释局部取舍，但不能成为报告主线。
- 不得编造完成情况、测试结果、排名、队员、指导老师、AI 使用情况或验证证据。
- 不要把私人 `etc/` 路径写成公开稳定来源，除非该段明确是在说明本地开发证据，
  且文字清楚交代了这个边界。

构建与验证：

```sh
make -C report
git diff --check -- report
```

对于新增且未跟踪的文本文件，还要使用 `git diff --no-index --check -- /dev/null
<file>` 检查空白问题。
